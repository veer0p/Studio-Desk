# Invoices API

Base path: `/api/v1/invoices`

Endpoints:
- `GET /api/v1/invoices`
- `POST /api/v1/invoices`
- `GET /api/v1/invoices/:id`
- `PATCH /api/v1/invoices/:id`
- `POST /api/v1/invoices/:id/send`
- `POST /api/v1/invoices/:id/payment-link`
- `POST /api/v1/invoices/:id/record-payment`
- `POST /api/v1/invoices/:id/credit-note`
- `GET /api/v1/invoices/view/:token`

Critical rules:
- Invoice amounts are immutable after creation. `PATCH` only accepts `notes`, `internal_notes`, and `due_date`.
- Invoice numbers are generated by the database trigger. The API never sends `invoice_number` in insert payloads.
- GST is auto-detected from state codes when `gst_type` is omitted, and is always split as `CGST+SGST` or `IGST`.
- Credit notes can only be created from paid invoices.
- Public invoice views never expose `internal_notes`, bank details, GSTIN, or `access_token`.
- Existing payment links are reused while still valid.

Template-free invoice creation:
- `invoice_type` supports `advance`, `balance`, `full`.
- `credit_note` invoices are created through `POST /:id/credit-note`.
- All amounts are stored and returned as numeric strings with two decimals.

Public invoice view:
- Authorized only by the 64-character token.
- Marks `viewed_at` asynchronously.
- Returns invoice summary, line items, client name, and safe studio branding fields.
