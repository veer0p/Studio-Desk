# StudioDesk Backend

## Purpose
StudioDesk is a Next.js 14 backend application for photography studio management. It provides a comprehensive SaaS platform handling leads, bookings, proposals, contracts, invoicing, payments, team scheduling, photo galleries, and client communication automation.

## Tech Stack
- **Framework**: Next.js 14.2.16 (App Router)
- **Database**: Supabase (PostgreSQL)
- **Validation**: Zod v4
- **Testing**: Vitest + jsdom
- **PDF**: @react-pdf/renderer
- **Email**: Resend
- **Payments**: Razorpay
- **Photo Storage**: Immich
- **Language**: TypeScript 5

## Key Files

### middleware.ts
Next.js edge middleware for authentication and CORS.
- **Purpose**: Runs on every request before the route handler.
- **Auth Flow**: Creates Supabase SSR client, retrieves user from cookies.
- **Protected Routes**: All `/api/*` and `/dashboard/*` paths require authentication.
- **Public Exceptions**: `/api/v1/webhooks/**`, `/api/v1/gallery/**/lookup`, `/api/v1/inquiry/**`, `/api/v1/portal/**`, `/api/v1/auth/**`, `/api/v1/admin/auth/**`, `/api/v1/ping`.
- **CORS**: Allows all origins, all standard HTTP methods, with credentials.
- **Response**: Returns 401 JSON for unauthenticated API requests, redirects to `/login` for dashboard routes.
- **Matcher**: Excludes `_next/static`, `_next/image`, `favicon.ico`.

### next.config.mjs
Next.js configuration file. Sets up module resolution, bundling, and app-specific settings.

### tsconfig.json
TypeScript configuration with path aliases (`@/*` → `./*`), strict mode enabled.

### package.json
- **Scripts**:
  - `dev`: Next.js dev server on port 3001 (IPv4-first DNS)
  - `build`: Production build
  - `test:unit`: Vitest unit tests
  - `test:api`: Vitest API tests
  - `test:integration`: Vitest integration tests
  - `test:deep`: Sequential integration tests with verbose output
  - `test:coverage`: Vitest with coverage
  - `db:seed`: Run main seed script
  - `db:seed:users`: Seed test users only
  - `db:verify:users`: Verify test users
  - `db:rollback:users`: Delete test users
  - `db:reset:seed`: Full DB reset + reseed

### vitest.config.ts / vitest.integration.config.ts
Vitest configurations for unit and integration test suites.

## Architecture

### Layer Pattern
```
Request → middleware.ts → API Route → Service → Repository → Supabase
                                    ↓
                              Validation (Zod)
                                    ↓
                              External Services (Razorpay, WhatsApp, Email, Immich)
```

### Module Relationships
- **API Routes** (`app/api/v1/**/route.ts`) → Call **Services** (`lib/services/*.service.ts`)
- **Services** → Call **Repositories** (`lib/repositories/*.repo.ts`) for DB operations
- **Services** → Use **Validations** (`lib/validations/*.schema.ts`) for input validation
- **Services** → Use **External Clients** (Razorpay, Resend, WhatsApp, Immich) for integrations
- **Admin Guards** (`lib/admin/guards.ts`) → Validate admin sessions via **AdminAuthService**

### Data Flow
1. Client sends HTTP request
2. `middleware.ts` checks auth (with public route exceptions)
3. API route handler extracts parameters
4. Service validates input with Zod schema
5. Service applies business rules
6. Service calls repository for DB operations
7. Service triggers side effects (notifications, audit logs) via fire-and-forget
8. Service returns response

### Security Patterns
- **Row-Level Security (RLS)**: Supabase RLS for tenant isolation (bypassed by admin client)
- **Encryption**: Bank account numbers, API keys encrypted via `@/lib/crypto`
- **Token-based Access**: 64-char hex tokens for contracts, invoices, galleries, invitations
- **Rate Limiting**: Inquiry forms, integration tests, gallery lookups
- **Session Management**: Admin sessions with 8h absolute expiry, 30min idle timeout
- **HTML Sanitization**: Contract content stripped of script tags
- **XSS Prevention**: Template variables sanitized, script tags removed from PDF output
